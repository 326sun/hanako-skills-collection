# Diagnose

这就是这个技能的灵魂。其他都是机械步骤。

## Phase 1 — 建立反馈回路

如果你有一个快速、确定性、agent 可运行的通过/失败信号，你就能找到根因。二分法、假设检验、工具化探测——这些都只是在消费这个信号。没有这个信号，盯着代码看一天也没用。

在这个阶段投入不成比例的精力。**激进。有创造力。拒绝放弃。**

### 构建反馈回路的 10 种方式——按优先级尝试

1. **失败测试**——在任意能触达 bug 的接缝上写单元测试、集成测试或端到端测试
2. **Curl/HTTP 脚本**——对着运行中的开发服务器发请求
3. **CLI 调用**——用固定输入运行，diff stdout 对比已知正确输出
4. **无头浏览器脚本**（Playwright/Puppeteer）——驱动 UI，断言 DOM/控制台/网络
5. **回放捕获的 trace**——把真实请求/事件日志存到磁盘，在隔离环境中重放
6. **一次性线束**——启动系统的最小子集（单个服务 + mock 依赖），用单个函数调用执行 bug 路径
7. **属性/模糊测试循环**——如果 bug 是"有时输出错误"，跑 1000 次随机输入找失败模式
8. **二分线束**——如果 bug 在两个已知状态之间出现（commit、数据集、版本），自动化"在状态 X 启动→检查→重复"以便 `git bisect run`
9. **差分循环**——用相同输入跑旧版本和新版本，diff 输出
10. **HITL bash 脚本**——最后手段。如果需要人工点击，用脚本驱动**人**的操作流程，让循环仍然结构化

### 迭代优化回路

有了基本回路后问自己：

- 能更快吗？（缓存 setup、跳过无关初始化、缩小测试范围）
- 信号能更尖锐吗？（断言具体症状，不是"没崩溃"）
- 能更确定吗？（固定时间、固定随机种子、隔离文件系统、冻结网络）

30 秒的不稳定回路只比没有回路好一点。2 秒的确定性回路是调试超能力。

### 非确定性 bug

目标不是完美复现，而是**提高复现率**。把触发条件循环 100 次、并行化、加压力、缩小时间窗口、注入 sleep。50% 概率的 bug 是可调试的；1% 不是——继续提高复现率直到可调试。

### 当你确实构建不了回路

停下来，明确说明你试了什么。向用户寻求：(a) 能复现问题的环境访问权限，(b) 捕获的产物（HAR 文件、日志转储、core dump、带时间戳的屏幕录像），或 (c) 在生产环境加临时探针的许可。

**不要在没有回路的情况下进入 Phase 2。**

## Phase 2 — 复现

跑回路。看着 bug 出现。

确认：
- [ ] 回路产生的是**用户描述的**失败模式——不是附近碰巧也失败的另一个问题。错误的 bug = 错误的修复。
- [ ] 失败可重复（或对非确定性 bug，复现率足够高以支持调试）
- [ ] 你捕获了确切的症状（错误信息、错误输出、慢速计时）以便后续阶段验证修复

不复现不前进。

## Phase 3 — 生成假设

在测试任何假设之前，生成 **3-5 个排序假设**。单假设生成会锚定在第一个看似合理的想法上。

每个假设必须是**可证伪的**：陈述它做了什么预测。

> 格式："如果 X 是原因，那么改变 Y 会让 bug 消失 / 改变 Z 会让 bug 更严重。"

如果你无法陈述预测，这个假设是感觉层面的——丢弃或磨尖它。

**测试前把排序列表给用户看。** 用户通常有领域知识可以瞬间重排（"我们刚对 #3 做了变更"）或知道哪些假设已经被他们排除过。这是一个廉价检查点。

## Phase 4 — 工具化探测

每次探测必须对应 Phase 3 中的具体预测。**一次只改变一个变量。**

工具优先级：
1. **调试器/REPL 检查**——一个断点胜过十行日志
2. **定向日志**——在区分假设的边界处精确打日志
3. 永远不要"全量日志然后 grep"

**为每个调试日志打唯一前缀标记**，如 `[DEBUG-a4f2]`。清理时一个 grep 搞定。无标记的日志是永久垃圾；有标记的日志会被清理。

**性能回归分支。** 对于性能回归，日志通常是错的工具。正确做法：建立基线测量（计时线束、`performance.now()`、profiler、查询计划），然后二分。先测量，再修复。

## Phase 5 — 修复 + 回归测试

**修复前写回归测试**——但只有当存在**正确的测试接缝**时才这样做。

正确的测试接缝是指：测试能覆盖**真实的 bug 模式**，和它在调用点实际发生的方式一致。如果唯一可用的接缝太浅（单调用者测试但 bug 需要多调用者才能触发，单元测试无法复现触发 bug 的调用链），那里的回归测试给的是虚假信心。

**如果不存在正确的测试接缝，这本身就是发现。** 标注它。代码库的结构使得某些 bug 无法被有效测试——这是架构问题，不是测试问题。

修复本身：
- 一次一个变更
- 不要顺手重构
- 不要搭车改其他东西
- 验证回归测试通过

## 3+ 次修复失败 = 架构问题

如果你尝试了 3 次以上修复仍未解决，停止。这不是假设错误，是架构错误。

信号：
- 每次修复揭示出不同位置的新的共享状态/耦合/问题
- 修复需要"大规模重构"才能实施
- 每次修复在其他地方产生新症状

**停下来和用户讨论架构。** 不要尝试第 4 次修复。

## 红牌——立即停止并回到 Phase 1

当你发现自己这样想：
- "快速修一下，之后再调查"
- "试试改 X 看看行不行"
- "一次改几处，跑测试"
- "跳过测试，我手动验证"
- "八成是 X，修了就知道了"
- "我不完全理解但这可能有用"
- "再试一次修复"（已经试过 2 次了）
- 每次修复在不同地方揭示新问题

**所有这些意味着：停下来。回到 Phase 1。**

## 禁止的行为

- 没有反馈回路就动手修
- 只生成一个假设就测试
- 同时改变多个变量
- 没有正确测试接缝时强写回归测试
- 3 次修复失败后继续尝试
- 在 RED 状态下重构

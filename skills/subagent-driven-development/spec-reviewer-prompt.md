# 规格审查子 Agent 派发模板

仅当实现子 Agent 报告 DONE 后派发。目的是验证实现者构建的正好是被要求的东西，不多不少。

## 核心原则

**不要相信实现者的报告**。实现者可能乐观、遗漏或误读规格。你必须独立验证一切。

不要：
- 相信他们说的实现了什么
- 相信他们对完整性的声明
- 接受他们对需求的解读

要做：
- 亲自读他们写的代码
- 逐行对照实际实现和需求
- 检查声称实现但实际缺失的部分
- 寻找他们未提及的多余功能

## Prompt 结构

```
## 被要求构建的东西

[粘贴完整的任务需求文本]

## 实现者声称构建的东西

[粘贴实现者的报告]

## 你的检查

阅读实现代码，验证：

**缺失的需求**：
- 所有被要求的东西都实现了吗？
- 有没有需求被跳过或遗漏？
- 有没有声称有效但实际未实现的东西？

**多余/不需要的工作**：
- 有没有构建未被要求的东西？
- 有没有过度设计或添加不必要的功能？
- 有没有添加规格之外的"顺手优化"？

**误解**：
- 有没有对需求的解读与预期不同？
- 有没有解决了错误的问题？
- 实现了正确的功能但用了错误的方式？

## 输出格式

✅ Spec compliant — 代码实际内容与规格完全匹配（不是相信报告，是读了代码后确认）

❌ Issues found:
- [文件:行号] 具体问题描述
```

## 关键规则

- 规格不合规 = 任务未完成。不要因为"差不多了"或"小问题"就放行
- 审查发现的问题必须由同一个实现子 Agent 修复
- 修复后必须重新审查，直到通过

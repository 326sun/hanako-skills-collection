# 代码质量审查子 Agent 派发模板

**仅在规格审查 ✅ 通过后派发**。目的是验证实现的质量：干净、可测试、可维护。

## 审查维度

使用 `code-reviewer` skill 的标准审查维度，额外关注：

- 每个文件是否有单一清晰职责和良好定义的接口？
- 单元是否被分解为可独立理解和测试的粒度？
- 实现是否遵循计划中的文件结构？
- 新增代码是否创建了已经很大的文件，或显著增大了已有文件？（不标记已有代码的问题，只看本次变更的贡献）

## Prompt 结构

```
## 审查任务

[任务摘要，来自实现者报告]

## 需求和计划

[任务需求文本]

## 变更范围

BASE_SHA: [任务开始前的 commit]
HEAD_SHA: [当前 commit]

## 审查输出

### Strengths（做得好的地方）

### Issues

#### Critical（必须修复）
- [文件:行号] 问题 → 建议

#### Important（应当修复）
- [文件:行号] 问题 → 建议

#### Minor（可选修复）
- [文件:行号] 问题 → 建议

### Assessment（整体评价）
✅ Approved / ❌ Changes Requested
```

## 关键规则

- 代码质量不通过 = 任务未完成
- 审查发现的问题由同一个实现子 Agent 修复
- 修复后必须重新审查，直到通过
- 实现者的自审查不能替代正式审查

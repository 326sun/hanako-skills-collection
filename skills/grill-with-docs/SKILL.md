# Grill With Docs

需求不对齐是最昂贵的 bug。你理解的和用户想要的是两回事。

这个技能做三件事：
1. 逐题追问，一次一个问题，直到你们对计划达成共享理解
2. 建立项目的共享词汇表（CONTEXT.md），消除术语歧义
3. 在真正需要的时候写架构决策记录（ADR），不做文档垃圾

## 核心纪律

**一次只问一个问题。等用户反馈后再问下一个。**

这意味着：不要一口气列五个问题让用户回答。逐个追问。每个问题基于前一个回答深化。

**如果可以查代码库来回答自己的问题，查代码库。不要问用户能从代码里读到的信息。**

这意味着：先探索代码结构和现有文档，再向用户提问。用户的时间比你的 token 预算贵。

## 流程

### Step 1: 探索现有文档

检查项目根目录是否存在 `CONTEXT.md` 或 `CONTEXT-MAP.md`。

如果 `CONTEXT-MAP.md` 存在，说明项目有多个上下文。`CONTEXT-MAP.md` 指向每个上下文的实际位置。根据你要修改的模块，找到对应的 `CONTEXT.md`。

如果没有任何上下文文件，记下这个事实——稍后你会创建第一个。

### Step 2: 展开审问

从设计树的根部开始，逐步向下。每条分支上的依赖关系逐个解决。

对每个决策点：

1. **挑战模糊术语。** 用户说"账号"时，追问：是 Customer 还是 User？两者是不同的东西。提出精确的规范术语。

2. **挑战词汇冲突。** 如果 CONTEXT.md 里已经把"取消"定义为 X，但用户现在说的"取消"明显是 Y，立刻指出来。

3. **用具体场景压力测试。** 当用户描述领域关系时，构造边界场景来逼迫精确化。例如："如果用户在支付进行到一半时关闭浏览器，订单状态应该是什么？"

4. **交叉验证代码。** 用户说某件事"就是这样工作的"，去代码里查。如果代码和描述矛盾，把矛盾摊开。

5. **每次提供一个推荐答案。** 你不是在问开放式问题——你是在提出你的判断并请求校准。"我建议把 X 定义为 Y，因为 Z。对吗？"

### Step 3: 更新 CONTEXT.md

不要在会话结束时批量写入。每当你和用户确定了一个术语的定义，立刻写进 CONTEXT.md。

CONTEXT.md 是纯词汇表，不包含：
- 实现细节
- 技术规格
- 临时的开发笔记

**只放术语定义**。如果一个新加入项目的工程师读了 CONTEXT.md 就能理解领域术语，那就是好词汇表。

格式：

```markdown
# Context

## [术语名]

[一句话定义]

### 与其他术语的关系
- 由 [X] 创建
- 被 [Y] 消费
- 与 [Z] 的区别是 [关键差异]
```

如果项目有多个上下文，使用 `CONTEXT-MAP.md` 来路由：

```markdown
# Context Map

- `src/ordering/CONTEXT.md` — 订单领域
- `src/billing/CONTEXT.md` — 计费领域
- `docs/adr/` — 跨领域架构决策
```

### Step 4: 按需写 ADR

只有同时满足以下三个条件时才写 ADR：

1. **难以逆转**——将来改变这个决定的成本是显著的
2. **没有上下文会困惑**——未来的读者看到代码会问"他们为什么这么做？"
3. **真正做了取舍**——确实有可行的替代方案，你选了其中一个有具体原因

缺少任何一个，跳过 ADR。

格式：`docs/adr/0001-简短标题.md`

```markdown
# ADR 0001: 标题

**状态**: 已采纳
**日期**: YYYY-MM-DD

## 背景

[为什么要做这个决定]

## 决策

[我们决定做什么]

## 替代方案

[我们考虑过但不选的方案，以及为什么不选]
```

## 示例对话

```
用户：我想加一个订单取消功能

你：好的。先确认一下——CONTEXT.md 里你的「订单」定义是"用户提交的购买请求"。取消是指整个订单取消，还是允许取消订单中的部分商品？这两条路径完全不同。

用户：整个订单取消

你：明白。那取消后库存怎么处理？在 CONTEXT.md 中我看到了「库存预留」这个术语——取消订单时，已预留的库存是释放回可售池，还是进入待审核状态？

用户：释放回可售池

你：好。如果订单包含已发货的商品呢？这时候取消是否仍然可行，还是只允许取消未发货的订单？

...依此类推
```

## 禁止的行为

- 一口气列出所有问题让用户填
- 问可以从代码库中直接读到的问题
- 接受模糊术语不做追问
- 把 CONTEXT.md 写成技术规格文档
- 写不需要的 ADR（三个条件缺一个就不要写）

# Test-Driven Development

测试应该描述系统做什么，不是怎么做。好的测试在重构后仍然存活，因为它不关心内部结构。

## 核心原则

**行为测试，不是实现测试。** 测试通过公共接口验证行为。一个读起来像规格说明的测试——"用户可以结账有效购物车"——告诉你系统有什么能力。这些测试在重构中存活。

**坏测试的标志**：重命名一个内部函数，测试就断了。代码行为没变，测试在测实现。

## 反模式：水平切片

**绝对不要**先写所有测试，再写所有实现。这叫水平切片——把 RED 理解成"写所有测试"，把 GREEN 理解成"写所有代码"。

水平切片产出的测试质量很差：
- 你在测你**想象的**行为，不是**实际的**行为
- 你在测数据结构和函数签名，而不是面向用户的能力
- 测试对真实变化不敏感——行为断了测试通过，行为正常测试失败

**正确做法：垂直切片。** 一个测试 → 一个实现 → 重复。每个测试建立在上一个循环学到的信息之上。

```
错误（水平）：
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正确（垂直）：
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## 工作流

### 1. 规划

动手写代码之前：

- [ ] 确认公共接口应该是什么样
- [ ] 确认哪些行为最重要，优先测试
- [ ] 识别深度模块的机会（小接口，深实现）
- [ ] 列出要测试的行为（不是实现步骤）
- [ ] 获取用户确认

问用户："公共接口应该是什么样？哪些行为最重要，优先测？"

**你不能测试一切。** 确认哪些行为真正重要。把测试精力集中在关键路径和复杂逻辑上。

### 2. 曳光弹

先写**一个**测试，确认**一件事**：

```
RED:   写第一个行为的测试 → 测试失败
GREEN: 写最少代码让测试通过 → 测试通过
```

这是你的曳光弹——证明路径从头到尾是通的。

### 3. 增量循环

对每个剩余行为：

```
RED:   写下一个测试 → 失败
GREEN: 最少代码通过 → 通过
```

规则：
- 一次一个测试
- 只写刚好让当前测试通过的代码
- 不要预判未来的测试
- 测试聚焦在可观察行为上

### 4. 重构

所有测试通过后再重构。**绝不**在 RED 状态下重构。先到 GREEN。

重构时注意：
- [ ] 提取重复
- [ ] 深化模块（把复杂度移到简洁接口后面）
- [ ] 应用 SOLID 原则（适用时）
- [ ] 每个重构步骤后跑测试

## 每周期检查清单

```
[ ] 测试描述的是行为，不是实现
[ ] 测试只使用公共接口
[ ] 测试在内部重构后仍然存活
[ ] 代码恰好满足当前测试，不多一行
[ ] 没有投机性功能
```

## 选择测试接缝

**单元测试**适用于纯逻辑：计算、转换、验证规则。

**集成测试**适用于跨越模块边界的路径：API → 服务 → 数据库。

**端到端测试**适用于用户可见的关键流程。只测试真正重要的端到端路径——它们慢且脆弱。

选择正确的接缝：如果 bug 需要多个调用者才能触发，单调用者单元测试给的信心是虚假的。直接注明"当前代码库没有正确的测试接缝"比写一个假的正向测试更有价值。

## 禁止的行为

- 先写所有测试再写所有实现（水平切片）
- 测试内部方法或私有函数
- 通过数据库直接查询来验证（应该走公共接口）
- 在 RED 状态下重构
- 为未来的需求写测试
- Mock 内部协作者——mock 模块边界，不是内部细节

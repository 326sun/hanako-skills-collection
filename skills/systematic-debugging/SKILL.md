# 系统调试法

随机修补浪费时间和制造新 bug。快速补丁掩盖底层问题。

**核心原则**：永远先找到根因再动手修复。修症状就是失败。

**铁律**：没有完成 Phase 1（根因调查），就不能提出修复方案。

## 四阶段流程

必须按顺序完成每个阶段，禁止跳步。

### Phase 1：根因调查

**在尝试任何修复之前**：

1. **仔细读错误信息**
   - 不要跳过错误或警告
   - 错误信息往往包含确切解决方案
   - 完整读完堆栈跟踪
   - 记录行号、文件路径、错误码

2. **稳定复现**
   - 能可靠触发吗？
   - 精确步骤是什么？
   - 每次都发生？
   - 如果不能复现 → 收集更多数据，不要猜

3. **检查近期变更**
   - 什么变了可能导致这个？
   - git diff、最近提交
   - 新依赖、配置变更
   - 环境差异

4. **多组件系统中的证据收集**

   当系统有多个组件（CI → 构建 → 签名，API → 服务 → 数据库）：

   **在提出修复之前，先加诊断插桩**：
   ```
   对每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置传播
     - 检查每层的状态

   运行一次收集证据，看问题出在哪一层
   然后分析证据确定失败的组件
   然后深入调查该组件
   ```

5. **追踪数据流**（详见 `root-cause-tracing.md`）
   - 坏的值的源头在哪里？
   - 谁用坏的值调用了这个？
   - 一直向上追溯，直到找到根源
   - 在根源处修复，不在症状处修复

### Phase 2：模式分析

**找到模式再修复**：

1. **找可工作的参照**
   - 同一代码库中类似的可工作代码在哪？
   - 什么东西和坏掉的类似但能正常工作？

2. **对比参照实现**
   - 如果实现某个模式，完整读完参照实现
   - 不要略读，读每一行
   - 在应用之前完全理解模式

3. **识别差异**
   - 可工作和不可工作之间有什么区别？
   - 列出每个差异，无论多小
   - 不要假设"这个不可能有影响"

4. **理解依赖**
   - 这个组件依赖什么其他组件？
   - 需要什么设置、配置、环境？
   - 它做了什么假设？

### Phase 3：假设与检验

**科学方法**：

1. **形成单一假设**
   - 清晰陈述："我认为 X 是根因，因为 Y"
   - 写下来
   - 具体，不模糊

2. **最小化测试**
   - 做最小改动来测试假设
   - 一次只改一个变量
   - 不要一次修多个东西

3. **验证后再继续**
   - 有效？→ 进入 Phase 4
   - 无效？→ 形成新假设
   - 不要在上面叠加更多修复

4. **不知道自己不知道时**
   - 直接说"我不理解 X"
   - 不要假装知道
   - 寻求帮助
   - 继续研究

### Phase 4：根本修复

**修复根因，不修症状**：

1. **创建失败测试用例**
   - 最简复现
   - 自动化测试（如果可能）
   - 没有测试框架时用一次性测试脚本
   - 修复前必须要有（使用 `test-generator` 生成测试）

2. **实现单一修复**
   - 解决已识别的根因
   - 一次只改一个东西
   - 不"顺手"改进
   - 不捆绑重构

3. **验证修复**
   - 测试通过了？
   - 没有其他测试被破坏？
   - 问题实际解决了？

4. **如果修复失败**
   - 停
   - 数一下：已经试了多少次？
   - < 3 次：回到 Phase 1，用新信息重新分析
   - **≥ 3 次：停，质疑架构（见第 5 步）**
   - 不要不经过架构讨论就尝试第 4 次修复

5. **3 次以上修复失败 → 质疑架构**

   以下模式表明架构问题：
   - 每次修复在系统不同位置揭示新的共享状态/耦合/问题
   - 修复需要"大规模重构"才能实施
   - 每次修复在别处产生新症状

   **停下来质疑基础**：
   - 这个模式从根本上合理吗？
   - 我们"纯粹靠惯性"在坚持吗？
   - 应该重构架构还是继续修症状？

   **在和用户讨论前不要尝试更多修复。** 这不是假设失败，这是架构错了。

## 红线——停下来走流程

如果发现自己在想：
- "先快速修一下，后面再调查"
- "试试改 X 看看行不行"
- "一次改多个东西，跑一下测试"
- "跳过测试，我手动验证"
- "大概率是 X，我直接修了"
- "我不完全理解但这样可能有用"
- "模式说 X 但我换个方式"
- "主要问题是：[在未调查的情况下列出修复方案]"
- "再试一次"（已经试了 2 次以上）
- 每次修复在不同位置发现新问题

**以上全部意味着：停。回到 Phase 1。**

## 常见借口揭穿

| 借口 | 现实 |
|---|---|
| "问题简单，不需要走流程" | 简单问题也有根因。流程对简单 bug 也很快。 |
| "紧急，没时间走流程" | 系统调试比瞎猜乱试更快。 |
| "先试这个，再调查" | 第一个修复定了调。从一开始就做对。 |
| "确认修复有效再补测试" | 没测试的修复站不住。先测试才能证明。 |
| "一次修多个省时间" | 无法隔离什么起了作用。还会造成新 bug。 |
| "参照太长，我改改用" | 部分理解保证出 bug。完整读完。 |
| "我看到问题了，直接修" | 看到症状 ≠ 理解根因。 |
| "再试一次"（2 次以上失败后） | 3 次以上失败 = 架构问题。质疑模式，别再修了。 |

## 速查表

| 阶段 | 关键活动 | 成功标准 |
|---|---|---|
| **1. 根因** | 读错误、复现、查变更、收集证据 | 理解了 WHAT 和 WHY |
| **2. 模式** | 找可工作参照、对比 | 识别差异 |
| **3. 假设** | 形成理论、最小化测试 | 确认或形成新假设 |
| **4. 修复** | 创建测试、修复、验证 | Bug 解决，测试通过 |

## 当流程揭示"没有根因"

如果系统调查表明问题确实来自环境、时序或外部因素：

1. 你已经完成了流程
2. 记录你调查了什么
3. 实现适当的处理（重试、超时、错误提示）
4. 添加监控/日志以供将来调查

**但是**：95% 的"没有根因"案例是不完整的调查。
